## Overview

Logs are time-stamped entries that track the history of individual rows. Each log entry captures a score (0.00 to 10.00), a comment, and optionally a specific event date. Logs enable you to monitor changes, track performance over time, and maintain a detailed history of your data.

<Info>
Think of logs as a journal or timeline for each row - recording observations, ratings, and notes as things evolve.
</Info>

## Data Structure

### Database Schema

```sql
CREATE TABLE log_entry (
  id            BINARY(16) NOT NULL,
  owner_id      VARCHAR(64) NOT NULL,
  row_id        BINARY(16) NOT NULL,
  score         DECIMAL(4,2) NOT NULL,
  comment       TEXT NOT NULL,
  event_date    DATE NULL,
  created_at    TIMESTAMP(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  PRIMARY KEY (id),
  CONSTRAINT fk_log_row
    FOREIGN KEY (row_id) REFERENCES object_row(id)
      ON DELETE CASCADE,
  KEY ix_log_owner_row_created (owner_id, row_id, created_at),
  CONSTRAINT ck_log_score_range
    CHECK (score >= 0.00 AND score <= 10.00)
);
```

<Note>
The score constraint ensures all ratings are between 0.00 and 10.00, providing a consistent 10-point scale.
</Note>

| `row_id` | UUID | Yes | 16 bytes | Parent row being tracked |
| `score` | Decimal | Yes | 0.00 - 10.00 | Rating on a 10-point scale |
| `comment` | Text | Yes | Unlimited | Detailed notes or observations |
| `event_date` | Date | No | YYYY-MM-DD | Optional date when event occurred |
| `created_at` | Timestamp | Yes (auto) | - | When log was created |

## Score System

Logs use a 10-point rating scale:

- **0.00 - 2.00**: Poor/Terrible
- **2.01 - 4.00**: Below Average
- **4.01 - 6.00**: Average/Okay
- **6.01 - 8.00**: Good
- **8.01 - 10.00**: Excellent/Outstanding

<Info>
Scores are stored as `DECIMAL(4,2)`, supporting precision up to two decimal places (e.g., 7.50, 9.25).
</Info>

## Creating Logs

### Validation

When creating a log entry, the service validates:

1. **Row exists** - The parent row must exist
2. **Score range** - Score must be between 0.00 and 10.00 (enforced by database constraint)
3. **Comment required** - Comment cannot be empty
4. **Event date format** - If provided, must be valid YYYY-MM-DD format

### Implementation

```java
public LogResponse create(UUID rowId, CreateLogRequest req) {
    // Ensure the row exists
    boolean exists = rowRepository.existsById(rowId);
    if (!exists) 
        throw new ConflictException("Row no encontrada");

    // Parse optional event date
    LocalDate eventDate = null;
    if (req.getEventDate() != null && !req.getEventDate().isBlank()) {
        try { 
            eventDate = LocalDate.parse(req.getEventDate().trim()); 
        } catch (Exception e) { 
            throw new ConflictException(
                "eventDate inválida (YYYY-MM-DD)"
            ); 
        }
    }

    // Create log entry
    LogEntryEntity e = new LogEntryEntity();
    e.setOwnerId(ownerId());
    e.setRowId(rowId);
    e.setScore(req.getScore());
    e.setComment(req.getComment().trim());
    e.setEventDate(eventDate);

    LogEntryEntity saved = logRepository.save(e);

    return new LogResponse(
        saved.getId(), 
        saved.getScore(), 
        saved.getComment(), 
        saved.getEventDate(), 
        saved.getCreatedAt()
    );
}
```

## Listing Logs

Logs for a specific row are retrieved in reverse chronological order (newest first):

```java
public List<LogResponse> list(UUID rowId) {
    return logRepository
            .findByOwnerIdAndRowIdOrderByCreatedAtDesc(ownerId(), rowId)
            .stream()
            .map(l -> new LogResponse(
                l.getId(), 
                l.getScore(), 
                l.getComment(), 
                l.getEventDate(), 
                l.getCreatedAt()
            ))
            .toList();
}
```

## Event Dates vs Created Dates

Logs support two different timestamp concepts:

### `created_at` (Required)

- Automatically set when the log is created
- Represents **when you recorded** the observation
- Cannot be modified
- Always includes time (with millisecond precision)

### `event_date` (Optional)

- Manually set by the user
- Represents **when the event actually happened**
- Can be different from `created_at` (e.g., logging a car viewing from yesterday)
- Date only (no time component)
- Format: `YYYY-MM-DD`

<Accordion title="Why two dates?">
This separation allows you to record events retroactively. For example:

- You viewed a car on March 15th but didn't log it until March 17th
- `event_date`: `2024-03-15` (when the viewing happened)
- `created_at`: `2024-03-17T20:30:00.000Z` (when you logged it)

This maintains chronological accuracy while preserving the actual timeline of events.
</Accordion>

## Cascade Deletion

Logs are automatically deleted when their parent row is deleted:

```sql
CONSTRAINT fk_log_row
  FOREIGN KEY (row_id) REFERENCES object_row(id)
    ON DELETE CASCADE
```

<Warning>
Deleting a row will permanently delete all associated log entries. This operation cannot be undone.
</Warning>

## Best Practices

### Consistent Scoring

Develop a personal scoring rubric and stick to it:

```
10.00 = Perfect, no issues whatsoever
8.00-9.99 = Excellent with minor imperfections
6.00-7.99 = Good, meets requirements
4.00-5.99 = Acceptable but has notable issues
2.00-3.99 = Poor, significant problems
0.00-1.99 = Terrible, deal-breaker issues
```

### Detailed Comments

Write thorough comments that your future self will understand:

- **Good**: "Apartamento limpio, buena iluminación natural, cerca del metro"
- **Better**: "Visitado el sábado a las 11am. Apartamento muy limpio, grandes ventanas orientadas al sur con mucha luz natural. A 5 minutos caminando de la estación de metro. Vecinos parecen tranquilos. Único punto negativo: cocina algo antigua pero funcional."

### Use Event Dates

Always set `event_date` when logging past events to maintain accurate timelines.

## Analytics Potential

With log data, you can build analytics features:

- **Average score per row** - Calculate mean score across all logs
- **Score trends** - Visualize how scores change over time
- **Best/worst rated items** - Rank rows by their latest or average scores
- **Event frequency** - Track how often you log updates for different rows
